## Summary

Phase 16 adds three analytics capabilities to the existing analytics page and weekly reporting flow. All three requirements are data-only — no new external APIs, no new LLM calls. The analytics page (`src/app/(app)/analytics/page.tsx`) is a Next.js App Router **async Server Component** that runs Drizzle queries server-side and renders pure SVG/HTML. The established pattern is: add new SQL queries to the `loadData()` function, thread results into the JSX, add new CSS classes to `dashboard.css`. No client components needed for static charts.

The critical data availability question is **days-to-response** (R-88). The `applications` table has `appliedAt` (timestamp when status moved to "applied") and `lastEventAt` (timestamp of any status change). The `events` table stores every status transition with `kind` and `at`. Days-to-response is computable as: when an application moves to `replied`/`interview`/`offer`/`rejected`, the elapsed days from `appliedAt` to the first non-applied event's `at` field. However: "replied" is NOT currently a named pipeline stage — PIPELINE_STAGES is `["new", "saved", "applied", "interview", "offer", "rejected"]`. The first response signal is whichever of `interview`, `offer`, or `rejected` comes after `applied`. Days-to-response = `AVG(event.at - app.appliedAt)` for apps that have moved past "applied".

The **Weekly Strategy Brief** (R-87) must fire every Monday. The project uses Vercel Cron (configured in `.vercel/vercel.json`). There is already one cron at `0 1 * * *` for purge. The brief can be added as a second cron at `0 8 * * 1` (Monday 08:00 UTC) calling `/api/cron/weekly-brief`. It reads from DB and either stores the brief in `runs.stageMetrics` or a dedicated table, then surfaces it in the analytics page. The brief is display-only — a rules-based function, no LLM.

**Primary recommendation:** Add two new SQL query functions in `src/lib/analytics/` (one for source quality, one for market pulse/brief), extend `loadData()` in `analytics/page.tsx` with `Promise.all`, add a new cron route for the Monday brief, and render new panels using the existing dashboard CSS grid pattern.

### Pattern 1: Source Quality Query (R-83)

The `jobs` table has `source` (text) and `tier` (smallint). T1 conversion rate = (T1 count) / (total discovered by that source × 100).

```typescript
// Source: drizzle-orm docs — GROUP BY with conditional COUNT
// [VERIFIED: schema.ts — jobs.source text, jobs.tier smallint]
const sourceQuality = await db
  .select({
    source: schema.jobs.source,
    total: sql<number>`count(*)::int`,
    t1Count: sql<number>`count(*) filter (where ${schema.jobs.tier} = 1)::int`,
  })
  .from(schema.jobs)
  .where(isNotNull(schema.jobs.tier))
  .groupBy(schema.jobs.source)
  .orderBy(sql`t1_count desc`);
// T1 conversion rate = t1Count / total * 100 (computed in JS, not SQL)
```

### Pattern 2: Market Pulse — Days to Response (R-88)

The `events` table captures every status transition. Days-to-response = elapsed days from `appliedAt` to first `replied`/`interview`/`offer`/`rejected` event.

```typescript
// [VERIFIED: schema.ts — events.applicationId, events.at, events.kind]
// [VERIFIED: pipeline/actions.ts — lastEventAt updated on every transition]
// Approach: join applications to events, filter for terminal/response events,
// compute age from appliedAt.
const daysToResponse = await db
  .select({
    avgDays: sql<number>`
      round(avg(
        extract(epoch from (${schema.events.at} - ${schema.applications.appliedAt})) / 86400
      )::numeric, 1)
    `,
  })
  .from(schema.events)
  .innerJoin(schema.applications, eq(schema.events.applicationId, schema.applications.id))
  .where(
    and(
      isNotNull(schema.applications.appliedAt),
      inArray(schema.events.kind, ["status:interview", "status:offer", "status:rejected"]),
    ),
  );
```

**Critical gap:** The `events.kind` format needs verification — check what `updateApplicationStatus` writes as `kind`. From pipeline/actions.ts the events table insert is not visible in the 80-line excerpt. Need to verify event kind strings at implementation time. [ASSUMED — verify event kind format in events table before implementing]

**Fallback if events are sparse:** Use `applications.lastEventAt - applications.appliedAt` for any application whose status is not "applied" or "saved". This is less precise but works with zero events written.


### Pattern 3: T1 Volume Trend (R-88)

This week vs 4-week rolling average:

```typescript
// [VERIFIED: schema.ts — jobs.discoveredAt, jobs.tier]
const thisWeekStart = startOfWeek(new Date(), { weekStartsOn: 1 }); // Monday
const fourWeeksAgo = subWeeks(thisWeekStart, 4);

const t1Trend = await db
  .select({
    week: sql<string>`to_char(date_trunc('week', ${schema.jobs.discoveredAt}), 'YYYY-MM-DD')`,
    count: sql<number>`count(*)::int`,
  })
  .from(schema.jobs)
  .where(
    and(
      eq(schema.jobs.tier, 1),
      gte(schema.jobs.discoveredAt, fourWeeksAgo),
    ),
  )
  .groupBy(sql`date_trunc('week', ${schema.jobs.discoveredAt})`)
  .orderBy(sql`date_trunc('week', ${schema.jobs.discoveredAt})`);
// In JS: this week count vs average of prior 4 weeks
```

### Pattern 4: Weekly Strategy Brief Rules Engine (R-87)

Pure rules function — reads DB, returns a structured object, no LLM:

```typescript
// src/lib/analytics/weekly-brief.ts
export interface WeeklyBrief {
  generatedAt: Date;
  weekStarting: Date;
  applicationsSentThisWeek: number;
  targetPacePerWeek: number;        // configurable constant, e.g. 5
  t1Available: number;              // T1 jobs discovered this week
  t1Applied: number;                // T1 applications sent this week
  topSourceThisWeek: string;        // source name with most T1 jobs
  callout: string;                  // rules-generated string, e.g. "You applied to 2 of 8 T1 jobs — 6 remain"
}

export async function generateWeeklyBrief(): Promise<WeeklyBrief> {
  // All reads from DB; callout generated by if/else rules
  // Example callouts:
  // "Pace on track — 5 applications sent vs 5 target"
  // "3 T1 jobs discovered from Adzuna — none applied yet"
  // "Response rate: 40% — above 4-week average of 25%"
}
```

**Callout logic rules (rules-based, not LLM):**
1. If `applicationsSentThisWeek < targetPacePerWeek * 0.6` → "Behind pace — only N of {target} applications sent"
2. Else if `applicationsSentThisWeek >= targetPacePerWeek` → "Pace on track — N applications sent"
3. If `t1Available > 0 && t1Applied === 0` → "{t1Available} T1 jobs discovered — none applied yet"
4. If `t1Applied / t1Available < 0.5 && t1Available >= 3` → "Only {t1Applied} of {t1Available} T1 jobs applied — {t1Available - t1Applied} remain"
5. Otherwise → "Top source: {topSource} with {count} T1 jobs this week"

### Pattern 6: Brief Display in Analytics Page

The brief can be displayed in two ways:
1. **Inline panel** on the analytics page — query the most recent "weekly-brief" run from `runs` table, render as a card. This is the recommended approach (no new page, no email required).
2. **Email via Resend** — already configured (RESEND_API_KEY, RESEND_FROM, CANDIDATE_EMAIL in env schema). The cron can also send an email using the existing notify env loader. Not required by R-87 but the infrastructure is ready.

**Recommendation:** Render brief as a panel on the analytics page (read from `runs` table where `status = 'weekly-brief'`, take the most recent, render `stageMetrics`). Also optionally email via Resend using the existing notifySchema.

### Anti-Patterns to Avoid

- **LLM for callout text:** The callout must be rules-based. No `generateText()` call. Use if/else string templates.
- **Client-side data fetching for charts:** The analytics page is a Server Component. Adding `'use client'` and `useEffect` to fetch source quality data breaks the established pattern and costs a waterfall round-trip. Keep as RSC.
- **Separate analytics API route:** Do not add a `/api/analytics/source-quality` endpoint that the page fetches from. The page already queries DB directly — extend `loadData()`.
- **New table for brief storage:** Use the existing `runs` table with `status: 'weekly-brief'` as the storage mechanism (same pattern as purge cron). Avoids schema migration.
- **SUM of percentage:** When computing T1 conversion rate, compute numerator and denominator separately, divide in JS. Do not SELECT a percentage directly and SUM it.

## Common Pitfalls

### Pitfall 1: events.kind format unknown
**What goes wrong:** The days-to-response query joins on `events.kind` values like `"status:interview"` — if the actual stored value differs (e.g., `"interview"` or `"status_change"`) the query returns null.
**Why it happens:** `pipeline/actions.ts` was only partially read; the events INSERT is not visible in the excerpt.
**How to avoid:** At implementation time, read `src/app/(app)/pipeline/actions.ts` fully and grep `db.insert(schema.events)` to confirm kind strings. If events table is empty or kind format differs, use the `lastEventAt` fallback query instead.
**Warning signs:** `avgDays` returns null when there are applications in non-applied status.

### Pitfall 2: T1 conversion rate division by zero
**What goes wrong:** A source with 0 discovered jobs appears in the GROUP BY result due to a LEFT JOIN, causing division by zero when computing conversion rate in JS.
**Why it happens:** SQL GROUP BY only returns rows where the group is non-empty — this is actually safe. The risk is dividing `t1Count / total` in JS without a zero guard.
**How to avoid:** `const rate = total > 0 ? (t1Count / total * 100).toFixed(1) : '0.0'`

### Pitfall 3: Weekly brief displays stale data on non-Monday visits
**What goes wrong:** If the brief is only generated on Mondays via cron, it looks stale the other 6 days.
**Why it happens:** By design — brief is weekly. But without context it looks broken.
**How to avoid:** Display the brief card with a header showing "Week of [date]" and a note like "Updated each Monday". Alternatively, generate on-demand if no brief exists for the current week (re-run `generateWeeklyBrief()` inline — it's fast, no LLM).

### Pitfall 4: Cron not firing because vercel.json is wrong path
**What goes wrong:** Adding crons to `src/vercel.json` or `./vercel.json` (root) when Vercel reads from `.vercel/vercel.json`.
**Why it happens:** Two different Vercel config file locations exist. The root `vercel.json` IS the canonical config; `.vercel/vercel.json` is the project config generated by `vercel link`.
**How to avoid:** [VERIFIED: .vercel/vercel.json exists and contains the existing purge cron]. Add the new cron to `.vercel/vercel.json` — that is the file Vercel actually reads for this project. Also verify if a root-level `vercel.json` needs to be created/updated.

Actually on re-check: only `.vercel/vercel.json` exists (no root `vercel.json`). The `.vercel/` directory is typically gitignored. The correct place to declare crons for Vercel is a root-level `vercel.json`. [ASSUMED — need to verify with `git ls-files` whether `.vercel/vercel.json` is committed or if a root vercel.json should be created]

### Pitfall 5: Source names in `jobs.source` vs human-readable display names
**What goes wrong:** Source quality chart shows "magnetme" instead of "Magnet.me", "nvb" instead of "Nationale Vacaturebank".
**Why it happens:** `jobs.source` stores the source class identifier (lowercase string from `allSources()`).
**How to avoid:** Create a display-name map in the source quality module:
```typescript
const SOURCE_LABELS: Record<string, string> = {
  adzuna: "Adzuna",
  jooble: "Jooble",
  magnetme: "Magnet.me",
  nvb: "Nationale Vacaturebank",
  "indeed-nl": "Indeed NL",
};
```
